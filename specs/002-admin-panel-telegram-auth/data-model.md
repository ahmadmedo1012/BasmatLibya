# Phase 1 Data Model — Premium Redesign, Telegram Login, and Owner Admin Panel

**Plan**: [plan.md](./plan.md) · **Spec**: [spec.md](./spec.md) · **Research**: [research.md](./research.md) · **Date**: 2026-06-04

This document defines **only the additive changes** on top of feature `001`'s schema (`specs/001-digital-footprint-analyzer/data-model.md`). All `001` tables (`source_categories`, `lookups`, `lookup_categories`, `findings`, `aggregated_results`, `rate_limit_counters`) keep their existing shape — see G10 in `plan.md`.

Convention: PKs are `uuid` server-generated by `gen_random_uuid()`. Timestamps are `timestamptz NOT NULL DEFAULT now()`. All FKs cascade per the rules listed under each table.

---

## New tables

### `users`

One row per Telegram-identified person. Created on first successful Telegram sign-in; never created by any other path.

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid PK` | Internal stable identifier. |
| `telegram_id` | `bigint NOT NULL UNIQUE` | Numeric Telegram user id. The natural key — duplicates are impossible by the unique constraint, satisfying FR-004. |
| `display_name` | `text NOT NULL` | `first_name [+ ' ' + last_name]` from the most recent Telegram payload. Updated on every sign-in. |
| `username` | `text NULL` | Telegram `@username`. Mutable on Telegram side; updated on every sign-in. Can be NULL (some accounts have none). |
| `avatar_url` | `text NULL` | `photo_url` from the most recent Telegram payload. Public Telegram-CDN URL. |
| `role` | `text NOT NULL CHECK (role IN ('owner','user')) DEFAULT 'user'` | Derived from `OWNER_TELEGRAM_ID` env var on each sign-in (FR-009 / FR-009a). Never user-set. |
| `status` | `text NOT NULL CHECK (status IN ('active','suspended')) DEFAULT 'active'` | FR-018 lifecycle. |
| `joined_at` | `timestamptz NOT NULL DEFAULT now()` |  |
| `last_seen_at` | `timestamptz NOT NULL DEFAULT now()` | Touched on every authenticated request; used for "last activity" sort in admin. |
| `suspended_at` | `timestamptz NULL` | Set when `status` transitions to `suspended`. NULL when reinstated. |

Indexes:
- PK on `id`.
- `UNIQUE (telegram_id)` — natural-key dedup.
- `(role, status)` — admin filter.
- `(last_seen_at DESC)` — admin sort.

Constraints / invariants (enforced in service layer, R-13):
- The owner role can only be granted when `telegram_id == OWNER_TELEGRAM_ID`.
- Self-suspension and self-removal are blocked at the service layer (FR-010).
- Removal cascades sessions and audit-log actor pointers per the FK rules below.

### `sessions`

One row per active or recently-revoked sign-in. The opaque cookie value is never stored — only its `sha256` hash.

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid PK` |  |
| `user_id` | `uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE` |  |
| `token_hash` | `text NOT NULL UNIQUE` | `sha256(cookie_value)`, base64url. Lookups by hash, never by raw token. |
| `csrf_token` | `text NOT NULL` | Per-session double-submit token (R-04). Returned by `/api/auth/me`; required as `X-CSRF` on state-changing fetches. |
| `issued_at` | `timestamptz NOT NULL DEFAULT now()` |  |
| `last_seen_at` | `timestamptz NOT NULL DEFAULT now()` | Touched on every authenticated request; idle expiry uses this. |
| `expires_at` | `timestamptz NOT NULL` | `issued_at + session_lifetime_days` (R-07 site setting). Refreshed on activity, capped at `issued_at + 90 days`. |
| `revoked_at` | `timestamptz NULL` | Set immediately on suspension, removal, sign-out, or manual revocation. A non-null value invalidates the session regardless of `expires_at`. |
| `revoke_reason` | `text NULL CHECK (revoke_reason IN (NULL,'sign_out','suspended','removed','manual','expired','rotated'))` | Used by `session.invalidated` event payload. |
| `client_signature` | `text NOT NULL` | `sha256(user_agent + coarse_ip)`; diagnostics only, never used for binding (R-04). |

Indexes:
- PK on `id`.
- `UNIQUE (token_hash)` — single lookup per request.
- `(user_id, revoked_at)` — find all active sessions for a user (suspension cascade, listings).
- `(expires_at)` — periodic prune of expired rows.

State transitions (enforced in service layer):

```
active ──► revoked  (sign_out | suspended | removed | manual | rotated)
active ──► expired  (now() > expires_at; pruner sets revoked_at + revoke_reason='expired')
```

A revoked session is never reactivated; sign-in always issues a new row.

### `ai_model_entries`

External AI provider models registered by the owner for result enrichment (R-05, R-06).

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid PK` |  |
| `provider` | `text NOT NULL CHECK (provider IN ('openai','anthropic','google','nvidia','openai_compatible'))` | R-05 union. New providers are added by union extension + new adapter. |
| `model_id` | `text NOT NULL` | The provider's model identifier (e.g. `gpt-4o`, `claude-sonnet-4-6`, `gemini-2.5-pro`, `nvidia/nemotron-…`). |
| `display_label` | `text NULL` | Optional human-readable label shown in the admin UI. Defaults to `provider/model_id` when null. |
| `credential_ciphertext` | `jsonb NOT NULL` | `{ iv, ciphertext, tag }`, AES-256-GCM, key from `MODEL_SECRET_KEY` (R-06). Never returned in any GET. |
| `credential_last_four` | `text NOT NULL` | Last 4 characters of the plaintext credential at write time. Display-only; safe to return. |
| `system_prompt` | `text NOT NULL DEFAULT ''` | FR-011: Q3 advanced customisation. |
| `temperature` | `numeric(3,2) NOT NULL DEFAULT 0.20 CHECK (temperature >= 0 AND temperature <= 2)` | FR-011: Q3. |
| `max_output_tokens` | `integer NOT NULL DEFAULT 1024 CHECK (max_output_tokens >= 16 AND max_output_tokens <= 32000)` | FR-011: Q3. Provider-specific upper bound enforced in adapter. |
| `extra_params` | `jsonb NOT NULL DEFAULT '{}'::jsonb` | Provider-specific knobs (e.g. `top_p`). Validated by the adapter on write. |
| `status` | `text NOT NULL CHECK (status IN ('active','inactive','invalid')) DEFAULT 'inactive'` | `active` is the row used for enrichment. `invalid` is set when last validation failed. |
| `is_active` | `boolean NOT NULL DEFAULT false` | At most one row may have `is_active = true` (partial unique index below). |
| `validated_at` | `timestamptz NULL` | Set on successful validation. NULL means never validated. |
| `last_validation_error` | `text NULL` | Coarse class (e.g. `auth_failed`, `model_not_found`). |
| `created_by` | `uuid NOT NULL REFERENCES users(id) ON DELETE RESTRICT` | The owner who created the row. RESTRICT prevents accidental owner removal while entries exist. |
| `created_at` | `timestamptz NOT NULL DEFAULT now()` |  |
| `last_updated_by` | `uuid NULL REFERENCES users(id) ON DELETE SET NULL` |  |
| `last_updated_at` | `timestamptz NOT NULL DEFAULT now()` |  |

Indexes:
- PK on `id`.
- `UNIQUE INDEX one_active_ai_model ON ai_model_entries (is_active) WHERE is_active = true` — at most one active row at any time (FR-013).
- `(provider, model_id)` — lookup and listing.
- `(status, last_updated_at DESC)` — admin listing.

Validation rules:
- On create or on transition to `active`, the service runs `AiModelClient.validate(...)` (R-05). Failure surfaces a designed Arabic inline error and the row is **not** marked `active`/`is_active=true` (FR-012).
- Deletion of the active row is blocked unless another row is designated active first, or `enrichment_enabled` is set to false (FR-013, edge-case in spec).

### `audit_log_entries`

Single source of every admin and auth event (R-08).

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid PK` |  |
| `actor_user_id` | `uuid NULL REFERENCES users(id) ON DELETE SET NULL` | NULL for events with no authenticated actor (e.g. anonymous sign-in failures). |
| `event_class` | `text NOT NULL CHECK (event_class IN ('auth','admin'))` |  |
| `event_subclass` | `text NOT NULL` | E.g. `sign_in_success`, `user_suspend`, `ai_model_activate`, `site_setting_update`. |
| `target_kind` | `text NULL` | E.g. `user`, `ai_model_entry`, `site_setting`. NULL for events without a target. |
| `target_id` | `text NULL` | Free-form (uuid or setting key). |
| `before_value` | `jsonb NULL` | Pre-mutation state. Sensitive fields replaced by `{ redacted: true }` at the choke-point. |
| `after_value` | `jsonb NULL` | Post-mutation state. Same redaction rules. |
| `request_signature` | `text NULL` | `sha256(user_agent + coarse_ip)`; diagnostics only. |
| `created_at` | `timestamptz NOT NULL DEFAULT now()` |  |

Indexes:
- PK on `id`.
- `(event_class, created_at DESC)` — admin Audit page default sort.
- `(actor_user_id, created_at DESC)` — "all actions by user X".
- `(target_kind, target_id, created_at DESC)` — "all actions on entity Y".

Retention: rows older than `audit_retention_days` (default 365, see R-08) are purged by a daily sweep.

Invariants (enforced in service layer):
- Every admin mutation appends exactly one row in the same DB transaction (G9). The service-layer choke-point `auditLog.append()` is the only path that writes to this table; routes never `INSERT` directly.
- Sensitive field redaction is applied at the choke-point before insert; there is no path where a plaintext credential reaches the row.

### `user_lookup_associations`

Connects a signed-in user to lookups they ran (R-14).

| Column | Type | Notes |
|--------|------|-------|
| `id` | `uuid PK` |  |
| `user_id` | `uuid NOT NULL REFERENCES users(id) ON DELETE CASCADE` |  |
| `lookup_id` | `uuid NOT NULL REFERENCES lookups(id) ON DELETE CASCADE` |  |
| `associated_at` | `timestamptz NOT NULL DEFAULT now()` |  |
| `hidden_by_user_at` | `timestamptz NULL` | Set when the user removes the entry from their history (R-14). The lookup row itself is unaffected. |

Indexes:
- PK on `id`.
- `UNIQUE (user_id, lookup_id)` — one association per (user, lookup).
- `(user_id, hidden_by_user_at, associated_at DESC)` — user's own history listing (filters out hidden rows for non-admin reads).
- `(lookup_id)` — admin "view a user's lookup history" reverse lookup.

Service-layer rules:
- Inserted by the lookup-creation path (`POST /api/lookups`) **only when** the request carries a valid session cookie. Anonymous lookups create no association — they remain v1-style anonymous (G2).
- `DELETE /api/me/history/:lookupId` sets `hidden_by_user_at` (soft-hide); the row itself is preserved so the owner's `FR-020` view remains complete.

## Modifications to `001` tables

### `lookups` — one new column

| Column | Type | Notes |
|--------|------|-------|
| `owner_user_id` | `uuid NULL REFERENCES users(id) ON DELETE SET NULL` | Set when a lookup is created by an authenticated user. NULL for anonymous lookups (preserves v1 behaviour). |

This column is denormalised — the same fact lives in `user_lookup_associations` — but it removes one join on the most common write path (lookup intake) and lets the rate limiter scope per-user counts cheaply.

No other column on any `001` table is altered, dropped, or renamed (G10).

## Relationships

```
                                                ┌──────────────────────┐
                                                │   site_settings      │
                                                └──────────────────────┘
                                                          ▲
                                                          │ last_updated_by
              ┌──────────────────────────────────────────┐│
              │                                          ││
              │                                          ▼│
              │                                 ┌──────────────────────┐
              │                                 │   audit_log_entries  │
              │                                 │  (actor_user_id)     │
              │                                 └──────────────────────┘
              │
              ▼
       ┌────────────┐         ┌──────────────────────┐
       │   users    │ ◄────── │      sessions        │
       │            │         └──────────────────────┘
       │            │         ┌──────────────────────┐
       │            │ ◄────── │  ai_model_entries    │  (created_by, last_updated_by)
       │            │         └──────────────────────┘
       │            │         ┌──────────────────────┐
       │            │ ◄────── │ user_lookup_assoc    │ ─────► lookups
       │            │         └──────────────────────┘
       │            │ ◄─────────────── lookups.owner_user_id
       └────────────┘
```

Cascade rules summary:
- Deleting a `users` row cascades `sessions` and `user_lookup_associations`, sets `lookups.owner_user_id`, `audit_log_entries.actor_user_id`, `ai_model_entries.last_updated_by`, `site_settings.last_updated_by` to NULL, and is **blocked** by `RESTRICT` on `ai_model_entries.created_by` (cannot remove a user while entries they created exist — caller must reassign or delete the entries first; the admin UI surfaces this as a designed Arabic error).
- Deleting a `lookups` row cascades `user_lookup_associations` (in addition to v1's existing cascades).

## Validation rules (cross-referencing requirements)

| Rule | Source |
|------|--------|
| Telegram payload HMAC and freshness verified before any user/session row is written | FR-002, R-01 |
| `users.role = 'owner'` only when `users.telegram_id = OWNER_TELEGRAM_ID` | FR-009 / FR-009a, R-03 |
| `users.telegram_id` is unique → no duplicate users from re-sign-in | FR-004 |
| Self-suspend and last-owner-removal blocked at service layer | FR-010 |
| `sessions.token_hash` unique; raw token never stored or logged | R-02, FR-030 |
| Suspension transactionally revokes every active session of the target | FR-018, R-13 |
| At most one `ai_model_entries.is_active = true` | FR-013 (partial unique index) |
| `ai_model_entries.credential_ciphertext` round-trips cleanly under `MODEL_SECRET_KEY`; plaintext never logged or returned | FR-014, FR-030, R-06, G8 |
| `site_settings.value` validated against the per-key `zod` schema before insert/update | FR-022, R-07 |
| Every admin mutation produces exactly one `audit_log_entries` row in the same transaction | FR-023, SC-009, R-08, G9 |
| `user_lookup_associations.hidden_by_user_at` is set on user-side history removal; the underlying `lookups` row is untouched | R-14 |
| All new columns/tables are nullable or have safe defaults; no v1 column is altered | G10, R-12 |

## Migration: `0002_admin_panel.sql`

Single forward-only migration. Order:

1. `CREATE TABLE users (…);`
2. `CREATE TABLE sessions (…);`
3. `CREATE TABLE ai_model_entries (…);`
4. `CREATE TABLE site_settings (…);`
5. `CREATE TABLE audit_log_entries (…);`
6. `CREATE TABLE user_lookup_associations (…);`
7. `ALTER TABLE lookups ADD COLUMN owner_user_id uuid NULL REFERENCES users(id) ON DELETE SET NULL;`
8. Indexes listed above (including the partial unique index `one_active_ai_model`).
9. `INSERT INTO site_settings (key, value) VALUES …` — one row per default in R-07.

Roll-back drops the six new tables and the `lookups.owner_user_id` column. v1 continues to function (G10).

## Out of scope, reserved by design

- **Multi-owner / staff roles**: spec is single-owner. The `role` enum can grow to `('owner','staff','user')` later without changing existing rows.
- **Per-tenant key derivation for `MODEL_SECRET_KEY`**: one tenant at v1.
- **Session-binding to device fingerprint**: `client_signature` is diagnostics-only; binding is a future hardening, not a v1 requirement.
- **Audit log streaming to external SIEM**: structured stdout logs (`001 R-11`) plus the in-DB audit table are sufficient for v1; export is additive.
